## 一、项目总览类

### ★ Q：请介绍一下你这个项目。

**思路**：一句话定位 + 三种模式 + 技术栈 + 数据成果，60 秒讲完。

我做了一个基于 RAG 和多智能体协作的智能研究助手。用户上传自己的文档，就能就文档内容提问。

它有三种模式：普通问答做快速检索问答；深度研究把复杂问题拆成子问题逐个研究再汇总；多 Agent 模式由检索、分析、写作、审核四个专职 Agent 协作，审核 Agent 对答案把关、不合格就打回重写，实现自我修正。

技术上用 DeepSeek 做对话、智谱做向量化、LangGraph 做编排，还做了错误重试、成本追踪、系统评测，最后部署上线了。

因为「让 AI 读懂特定文档并回答」是企业最主流的 AI 落地场景，RAG 是它的核心。我想通过一个完整项目，把 Agent 开发的核心能力——检索、推理、多 Agent 协作、工程化——都覆盖到。研究助手这个场景正好能串起这些技术点，而且做出来是真能用的东西，不是玩具。

### Q：这个项目最难的地方是什么？怎么解决的？

**思路**：讲真实的、有技术含量的难题（用你真踩过的坑）。

另一个是回答质量：单次检索对跨章节的复杂问题会遗漏信息。我用任务拆解让每个子问题独立检索，又加审核 Agent 识别过度推断，把完整度和严谨度都提上去了，都有对比数据支撑。

## 二、RAG 相关（最高频）

### ★ Q：什么是 RAG？为什么需要它？

RAG 是检索增强生成。大模型的知识来自训练数据，不知道你的私有文档、也不知道训练之后的新事。RAG 就是回答前先从你的文档里检索相关内容，连同问题给模型，让它基于这些内容作答。

打个比方：不用 RAG 像闭卷考试只能靠记忆；RAG 像开卷——答题前先翻参考书。这样它能回答只有你文档里才有的问题，还能减少幻觉。

---

### ★ Q：RAG 的完整流程是怎样的？

两个阶段。**准备阶段**（离线一次）：文档切成小块，每块用 embedding 转成向量，存进向量库。

**查询阶段**（每次提问）：问题也转成向量，在库里找最相似的几块，把这些块和问题拼成提示词给模型，模型基于它们生成回答。

核心是「切块 → 向量化 → 存储 → 检索 → 生成」。

---

### ★ Q：为什么要切块？块大小怎么定？

两个原因：模型一次能处理的内容有限，不能塞整篇；检索时只想要相关那几段，切块让检索更精准。

块大小是权衡：太小容易切断完整信息、丢上下文；太大会夹带无关内容、稀释重点。我实测过不同 chunk_size 对命中率的影响，用数据选了较优值，不是拍脑袋。

---

### ★ Q：向量（embedding）是什么？为什么能做检索？

embedding 是把文字转成一串数字（向量），关键特性是：意思相近的文字，向量也相近。「猫」和「小猫」的向量接近，「猫」和「汽车」相差很远。

于是「两段文字意思像不像」就变成「两个向量近不近」，计算机算这个又快又准。检索时把问题向量和每个块向量算相似度，最相似的就是语义最相关的内容。

> **加分**：这就是为什么 RAG 能「按意思检索」——我问「差旅费」，文档写「差旅报销」，字面不同但向量相近，照样检索到。

---

### Q：余弦相似度是什么？你为什么自己实现？

余弦相似度衡量两个向量方向的接近程度，值在 -1 到 1，越接近 1 越相似，看方向不看长度，很适合比文本语义。

我自己实现是因为当时向量库底层库在我环境里有兼容问题，就用纯 Python 手写了：问题向量和每个块向量算余弦相似度、排序取最高的几个。这让我彻底理解了向量库内部就是「算相似度 + 排序」的高效版。

```python
def cosine_similarity(a, b):
    dot = sum(x*y for x, y in zip(a, b))
    na = sum(x*x for x in a) ** 0.5
    nb = sum(y*y for y in b) ** 0.5
    return dot / (na * nb)
```

---

### ★ Q：RAG 效果不好，你怎么排查？

**非常高频，答好体现工程思维。**

先分清是「检索问题」还是「生成问题」，方法是把检索到的内容打印出来看：

- 检索到的块里没有答案 → 检索问题，调 chunk_size、增大 top-k、改进 embedding。
- 块里有答案但模型没答对 → 生成问题，优化提示词。

我实际遇到过：一个问题答漏了，打印检索结果后发现关键信息没被捞到，定位是检索问题，调大 chunk_size 和 top-k 后解决。检索命中率和回答准确率两个指标分开看，就能定位瓶颈在哪一环。

---

### Q：怎么防止 RAG 幻觉（编造）？

- 提示词明确「只根据资料回答，没有就说不知道」。
- 提高检索质量，确保喂给模型的资料确实相关。
- 我还加了审核 Agent 检查有没有超出资料的编造。它实际抓出过一次过度推断——把「研发部门有奖金」错套到试用期新人，识别后打回重写了。

---

### Q：top-k 是什么？设大设小有什么影响？

top-k 是检索时取最相关的几块。太小可能漏掉相关信息（复杂问题尤其）；太大会塞进无关内容、增加成本、稀释重点。一般 3-5 起步，我根据评测调过这个值。

---

### Q：如果文档特别多（上百万条），你的检索还行吗？怎么优化？

我现在用的是纯 Python 遍历算相似度，小规模够用，但百万级会慢。要优化的话，用专业向量数据库（如 FAISS、Milvus），它们用近似最近邻（ANN）算法，能在海量向量里快速检索，不用逐个比。

> 这题答出「我知道我的方案的规模边界，也知道工业级方案」，很加分。

## 三、Agent 与多 Agent 相关

### ★ Q：什么是 AI Agent？和普通调用大模型有什么区别？

普通调用大模型是「一问一答」：给个输入，返回个输出，就结束了。Agent 则能「自主完成任务」：它会思考、调用工具、观察结果、再决定下一步，循环往复直到完成目标。

核心区别是 Agent 有「行动能力」和「循环」——它不只生成文本，还能用工具（搜索、计算、读写文件）、能根据结果调整策略。比如我的助手会自己判断问题难易、决定要不要拆解、协调多个子 Agent，这些都是 Agent 的自主性。

---

### ★ Q：讲讲你的多 Agent 系统是怎么协作的？

我设计了四个专职 Agent：检索 Agent 从知识库找资料，分析 Agent 提炼要点，写作 Agent 组织成答案，审核 Agent 检查质量。它们像流水线一样接力。

关键是审核环节有反馈回环：写作出稿后，审核 Agent 判断合不合格，不合格就带着修改意见退回让写作 Agent 重写，最多重试几次。这让系统能自我修正。

每个 Agent 用各自专属的角色提示词，专注做好一件事，比让一个提示词同时干检索、分析、写作，质量高得多。

---

### ★ Q：为什么用多个 Agent，而不是一个更强的提示词？

因为「分工专精」。让一个提示词同时扮演检索员、分析师、撰稿人、审核员，角色混杂，每个都做不精。拆成多个专职 Agent，每个只关注一件事、用最贴合的提示词，整体质量更高。

而且多 Agent 更灵活可扩展——加一个新角色（比如审核）只需加一个 Agent，不动其他的。我实测过多 Agent 比单 Agent 在复杂问题上回答更完整、更严谨。

---

### Q：你说的「自我修正 / Reflection」是什么？

就是让系统自己检查、自己改进。我的审核 Agent 读「资料 + 答案」，判断答案有没有编造、是否完整、是否条理清晰，返回通过或打回。打回时把问题反馈给写作 Agent 重写。这个「产出 → 审核 → 不合格就改」的循环，就是 Reflection。

它实际帮我抓出过一次过度推断并修正了。这是当前 Agent 领域的前沿方向，能有效降低错误。

---

### ★ Q：任务拆解（Planning）是怎么做的？

对复杂问题，我先让模型把它拆成 2-4 个子问题，输出成 JSON 数组，程序解析后逐个走 RAG 回答，最后把所有发现汇总成最终答案。

好处是每个子问题能独立检索，避免「一次检索顾此失彼」。比如「对比 A 和 B」，拆成「A 的情况」「B 的情况」「两者对比」分别查，比一次查全面得多。

> **追问「拆解失败怎么办」**：我做了兜底——JSON 解析失败就退回当作单个问题处理，不会崩。

---

### Q：你的智能路由是怎么实现的？

我用 LangGraph 搭了一张图：先有个判断节点，让模型判断问题是简单还是复杂（只回答 simple 或 complex）；然后用条件边分流——简单的走普通问答，复杂的走深度研究。这样简单问题秒答、复杂问题才深挖，兼顾速度和质量。

这解决了一个我发现的问题：无脑用深度研究，简单问题会被过度拆解、慢好几倍还没必要。

---

### Q：LangGraph 是什么？为什么用它？

LangGraph 是构建 Agent 工作流的框架，核心概念是节点（做一件事的步骤）和边（决定下一步去哪）。它特别适合表达「有分支、有循环」的复杂流程。

我用它做智能路由——判断节点 + 条件边分流。相比一堆嵌套 if-else，用图来表达流程更清晰、更好维护、也更好扩展。

---

### Q：Agent 的「记忆」是怎么实现的？大模型不是没记忆吗？

对，大模型本身没记忆，每次调用只能看到当次传入的内容。所谓「记忆」，就是每次都把该记住的信息重新塞进去。

我做了两层：短期记忆就是把对话历史一直放在消息列表里，每次带上，支持多轮追问；长期记忆则是把重要信息当文档存进向量库，需要时检索回来——本质就是把 RAG 用在记忆场景。

> **追问「短期记忆的局限」**：对话太长会超出上下文窗口。我用 max_turns 只保留最近几轮，更彻底的做法是把旧对话摘要或转入长期记忆。

## 四、工程实践类

### ★ Q：你怎么保证系统的稳定性？

我的助手依赖多个外部 API，任何一个抖动都可能出问题，所以做了几层防护：

- **重试**：API 偶发失败自动重试几次，扛过网络抖动。
- **超时**：请求设超时，避免卡死干等。
- **兜底**：文档某个加载失败就跳过不中断；问答出错给用户友好提示而不是崩。
- **防御性编程**：JSON 解析、模型返回都做了容错，解析失败有退路。

这些让系统从「能跑的 demo」变成「意外下也不崩的可靠系统」。

---

### ★ Q：你怎么评测系统效果？

我搭了一个评测集——一批「问题 + 标准答案」，覆盖多主题、多难度，还包含「文档里没有」的抗幻觉题。然后测两个核心指标：检索命中率（检索有没有捞到含答案的块）、回答准确率（答案对不对）。

判分我用了 LLM-as-judge——让另一个模型判断回答和标准答案是否语义一致，比简单的关键词匹配准。基于这些数据，我还做了 chunk_size 参数对比，数据驱动地优化。

> **追问「LLM 判分可靠吗」**：不是 100% 可靠，但比关键词匹配好得多，是业界主流方法。关键场景我会人工抽查校验。

---

### Q：你怎么控制成本？

我做了 token 和成本统计——每次模型调用后记录输入输出 token，按官网价格换算成钱，界面上实时显示。这样能看到哪种模式贵、贵在哪。

比如多 Agent 调用次数多、更贵，深度研究因为多次调用也贵，普通问答最便宜。有了数据，就能根据场景选合适的模式，做质量和成本的权衡。

---

### Q：你怎么定位性能瓶颈？

我做了分段计时，分别记录检索耗时和模型调用耗时。这样能看出瓶颈在哪一步。实测发现瓶颈主要在模型调用（网络 + 生成），检索很快。基于这个，优化方向是减少不必要的调用次数、或对简单问题走更轻的路径——这也是我做智能路由的原因之一。

> **核心观点**：没有测量就没有优化。先测，再定位，再优化。

---

### Q：你的项目是怎么组织代码的？

我按「关注点分离」拆成多个模块：config 管配置、loader 管文档加载、rag 管检索、planner 管拆解、assistant 管问答、agents 管多 Agent、graph 管路由、memory 管记忆、logger/metrics 管日志和统计、app 是界面。

每个文件职责单一，改检索只动 rag、加界面只调用已有函数不改逻辑。这让项目好维护、好扩展，别人也容易看懂。

---

### Q：部署时密钥怎么管理？

绝不把密钥写进代码或提交到 GitHub——公开仓库的密钥几分钟就会被爬虫扫到盗刷。本地用 `.env` 文件（被 `.gitignore` 排除），云端用平台的 Secrets 功能单独配置。我写了个小函数让同一份代码本地读 `.env`、云端读 secrets，两边都能跑。

> 安全意识是工程素养的重要一票，这题一定要答好。

## 五、概念基础类

### Q：大语言模型（LLM）的基本原理是什么？

简单说，LLM 是在海量文本上训练的神经网络，核心能力是「预测下一个词」。给它一段文字，它根据训练中学到的语言规律，一个词一个词地预测接下来最可能出现什么，连起来就是回答。

它没有真正的「理解」或「记忆」，但通过在极大规模数据上学习，涌现出了很强的语言、推理、知识能力。它的知识固定在训练时刻，这也是为什么需要 RAG 来补充实时和私有信息。

---

### Q：什么是 Token？

Token 是模型处理文本的基本单位，可以理解为「词的碎片」。一段文字会被切成若干 token，模型按 token 计算和计费。中文大致一个字 1-2 个 token。上下文窗口的大小、API 计费，都是按 token 算的。

---

### Q：什么是上下文窗口（context window）？

是模型一次能处理的最大 token 数，是模型的硬限制。所有你给它的内容——系统提示、对话历史、检索到的资料、当前问题——都要塞进这个窗口。超了就会报错或被截断。

这也是短期记忆的天花板：对话太长、历史累积太多，会撑爆窗口。所以要么控制历史长度，要么用长期记忆把信息存到外部、按需检索。

---

### Q：什么是提示工程（Prompt Engineering）？举个你项目里的例子。

就是通过设计提示词来引导模型给出更好的输出。同样的任务，提示词写得好不好，效果差很多。

我项目里的例子：一开始模型对「入职两年年假」这种需要简单推理的问题答「资料没提到」，我在提示词里加了「如果资料里能推断出答案，请合理推断」并给了个例子，它就能正确推理了。还有让审核 Agent 只回答 PASS/FAIL、让判断节点只回答 simple/complex，都是用提示词约束模型输出固定格式，方便程序处理。

---

### Q：什么是模型幻觉（Hallucination）？

指模型生成看似合理、实则错误或编造的内容。因为它本质是「预测下一个词」，有时会「一本正经地胡说」，比如编造不存在的事实、数据、引用。

RAG 是缓解幻觉的重要手段——让它基于真实资料回答。我还用审核 Agent 检查有没有超出资料的编造。但幻觉不能完全消除，所以关键场景要有人工校验。

---

### Q：temperature 参数是什么？

控制模型输出的随机性。低（接近 0）输出更确定、更保守，适合要准确、要稳定的任务（比如判断、抽取）；高则更多样、更有创造性，适合写作、头脑风暴。做 RAG 问答、判断这类要准确的任务，一般用较低的 temperature。

---

### Q：为什么用 DeepSeek 和智谱，而不是 OpenAI？

主要是国内访问稳定、成本低、中文效果好。DeepSeek 做对话性价比高，智谱提供 embedding 接口。我的架构是解耦的——对话模型和 embedding 模型可以独立替换，换成别的（比如 OpenAI）也只需改配置，不影响其他代码。

> 追问时强调「可替换性」，体现你的架构设计意识。

---

### Q：Fine-tuning（微调）和 RAG 有什么区别？什么时候用哪个？

微调是用特定数据继续训练模型，改变模型本身的参数，让它学会某种风格或专业知识。RAG 不改模型，而是在推理时检索外部知识喂给它。

RAG 适合「知识经常更新、需要引用来源、私有文档问答」的场景，成本低、更新快。微调适合「让模型学会某种固定风格、格式、或深度专业能力」的场景。很多实际系统两者结合用。我的项目是文档问答，知识需要灵活更新，所以用 RAG。

## 最后：面试心态与收尾

**讲项目讲成「故事」**：面试官爱听「我遇到什么问题 → 怎么分析 → 怎么解决 → 结果如何」的故事。你这 20 天有大量这样的真实素材：torch 崩溃换零依赖方案、审核 Agent 抓出过度推断、检索遗漏靠拆解解决……把它们讲成故事，比背概念动人。

**诚实面对不会的**：被问到不懂的，别硬编。可以说「这块我还没深入，但我的理解是……」或「我会这样去查证」。诚实 + 有思路，比不懂装懂强。

**用数据和 Demo 说话**：能亮数据就亮数据，能演示就演示。一个能点开的 Demo + 一张评测表，胜过千言万语。

**反问也是加分项**：面试官问「你有什么想问的」时，别说没有。问问团队在做什么 Agent 方向、技术栈、挑战——体现你的兴趣和思考。
