# my-agent-lab · 七天深度改造方案

> 目標:把現有的「教學級 RAG + 多智能體 demo」升級為 **工程級、評測驅動、接近生產形態** 的作品。
> 主線:打地基 → 檢索基建 → 檢索質量 → 真編排 → 評測驅動 → 可觀測與穩健 → 封裝與敘事。

---

## 現狀盤點(改造的起點)

| 維度 | 現況 | 主要短板 |
|---|---|---|
| 檢索 | `rag.py` 手寫餘弦 + Python list 全量掃描 | 無向量庫、無持久化、每次啟動重嵌入全庫、無混合檢索/重排/引用 |
| 編排 | `graph.py` 只做 3 節點路由;`deep_research`/`research_team` 是普通迴圈 | 沒有真正的狀態圖、並行、迭代檢索、檢查點、串流 |
| 評測 | `evaluate.py` 僅 LLM-judge 準確率 + 檢索命中率 | 無 faithfulness / context precision·recall、無回歸基線、無失敗歸因 |
| 工程 | `calculator` 用 `eval()`;`config.py` 重複 import;`planner.py` 字串硬解 JSON | 無測試、無 CI、無 tracing、無型別化配置 |
| 記憶 | `memory.py` 短期記憶;`day7` 有雛形長期記憶 | 未整合進主系統、無持久化 |

**貫穿七天的驗收標準**:每天結束時,`evaluate.py` 能跑出可比的數字,並把結果存進版本化報告,讓「深度」是可量化的,而不是口號。

## Day 1 — 打地基:修 Bug + 工程化底座

**目標**:先把地基修平,讓後面六天的改動可測試、可回歸。

**具體任務**
- 修真 Bug:`config.py` 刪掉重複的 `import os`;核對 `CHAT_MODEL = "deepseek-v4-flash"` 是否為有效模型名;`CHUNK_SIZE=159` 先改回合理值(如 500,Day 2 再調)。
- 安全:把 `day2/day3/day4` 與工具鏈裡的 `calculator` 從 `eval()` 換成 `ast.literal_eval` + 白名單運算,或用 `numexpr`,消除代碼注入。
- 型別化配置:引入 `pydantic-settings`,把散落的模型名、`TOP_K`、`CHUNK_*`、金鑰集中成一個 `Settings` 物件,啟動時做校驗(缺金鑰直接報錯而非 print 警告)。
- 測試骨架:建 `tests/`,用 `pytest` 為 `utils.retry`、`memory.ConversationMemory`、`cosine_similarity`、`loader.clean_text` 寫單元測試。
- 規範:加 `.env.example`、鎖版本、接入 `ruff`(lint)+ `black`(格式化)。

**深度體現**:從「能跑的腳本」變成「可維護、可測試的工程」——這是面試官第一眼看倉庫時的信號。

**交付與驗收**:`pytest` 全綠;`ruff check` 無報錯;配置集中且啟動即校驗。

---

## Day 2 — 把玩具 RAG 換成真正的向量庫 + 持久化 + 快取

**目標**:消滅「每次啟動重嵌入全庫、線性掃描」的根本性能與成本問題。

**具體任務**
- 用 **Chroma(持久化)** 或 **FAISS** 替換 `rag.py` 裡的 list + 手寫餘弦,索引落盤,第二次啟動秒開。
- 加 **embedding 快取**:對 chunk 文本做 hash,已算過的向量直接讀快取,大幅降本降延遲。
- 給每個 chunk 掛 **metadata**(來源檔名、chunk_id、頁碼)——這是 Day 3 引用溯源的前提。
- 重調切塊策略:合理的 `chunk_size/overlap`,對結構化文檔用 markdown/標題感知切分,而非純字元數。
- 把 `build_knowledge_base` 改成「檢測到已有索引則加載,否則構建」的冪等邏輯。

**深度體現**:真正的檢索基建 + 成本/延遲意識(可觀測的 embedding 呼叫次數與耗時)。

**交付與驗收**:重啟不重嵌入;`evaluate.py` 的檢索命中率不下降;記錄改造前後的啟動耗時對比。

---

## Day 3 — 檢索質量升級:混合檢索 + 重排 + 引用溯源

**目標**:把「能檢到」升級成「檢得準、答得有據」。

**具體任務**
- **混合檢索**:`rank-bm25`(稀疏)+ 向量(稠密),用 RRF(Reciprocal Rank Fusion)融合排序。你的測試集裡本來就有 BM25 這題,正好呼應。
- **重排(Rerank)**:在 top-N 上加一層 cross-encoder(如 `bge-reranker`)或 LLM 重排,顯著提升 top-k 精度。
- **查詢改寫 / 多查詢擴展**:對複雜問題生成多個檢索式,合併去重再檢索。
- **帶引用回答**:回答時輸出 `[來源: 檔名#chunk_id]`,把「主張 → 出處」對應起來;無出處的主張標記為推斷。這是最直接的抗幻覺手段。

**深度體現**:現代 RAG 完整棧(hybrid + rerank + grounding),而非單次稠密檢索。

**交付與驗收**:同一批複雜問題,檢索命中率與答案完整度較 Day 2 提升;答案可點到出處。

---

## Day 4 — 智能體編排真正上 LangGraph:狀態 / 並行 / 迭代檢索 / 人類介入

**目標**:把 `deep_research` 與 `research_team` 從普通迴圈,重寫成真正的狀態圖。

**具體任務**
- 用 **帶型別的 State** 重建圖:把「檢索 → 分析 → 寫作 → 審核(帶反饋回環)」建成節點與條件邊,而不是 `agents.py` 裡的 for 迴圈。
- **並行子問題**:`deep_research` 的子問題用 fan-out/fan-in 並行執行,而非順序 loop——延遲從線性降到近似單題。
- **迭代 / Agentic 檢索**:加入「檢索 → 推理 → 判斷是否需要再檢索」的循環,而不是一次檢索定生死。
- **檢查點(Checkpointer)**:接 LangGraph 的 thread 持久化,支持中斷續跑與多輪追問。
- **人類介入節點(可選)**:在寫作前插入審批節點;把中間步驟 **串流** 到 Streamlit。

**深度體現**:真正的 agentic orchestration(狀態、並行、迭代、可恢復),這是與「教學路由」拉開差距的核心。

**交付與驗收**:複雜問題耗時因並行明顯下降;圖可視化(LangGraph 自帶)能畫出來放進 README。

---

## Day 5 — 評測驅動:RAG 專用指標 + 回歸基線 + 失敗歸因

**目標**:讓「深度」可量化——這是 AI 專案最強的認真信號。

**具體任務**
- 升級 `evaluate.py`,加 RAG 專用指標:**faithfulness(忠實度)、context precision、context recall、answer relevancy**,以及 **引用準確率**(用 RAGAS 或自建帶評分細則的 LLM-judge)。
- **回歸框架**:一鍵跑三種模式,把結果寫成版本化報告(JSON + Markdown 表 + 圖表),自動對比與上一次基線的差值(Δ)。
- **測試集升級**:在現有 60 題基礎上補齊難度分層 + 對抗性/誘導幻覺題 + 每題的 **標準出處 span**,讓引用指標可算。
- **失敗歸因**:自動把錯題分類為「檢索未命中 / 綜合出錯 / 幻覺」,輸出歸因分布表。

**深度體現**:評測嚴謹度 + 可回歸性——把每一次改動都變成一組能拿去講的數字。

**交付與驗收**:一條命令產出完整評測報告;README 的效果表全部替換為真實數據(現在是佔位符)。

---

## Day 6 — 可觀測性 + 穩健性 + 性能

**目標**:讓系統「看得見、扛得住、跑得快」。

**具體任務**
- **Tracing**:接 LangSmith(或 OpenTelemetry),為每次請求生成 trace;把 `trace_id` 貫穿 `logger` 的每條日誌,能端到端還原一次問答。
- **守衛(Guardrails)**:輸入校驗;用 pydantic 校驗 `planner`/router 的結構化輸出(取代現在的字串硬解 JSON);超時/退避;失敗優雅降級。
- **性能**:對可安全快取的 LLM 答案加快取;把多次獨立 LLM 呼叫改成 **async/並發**,再砍一輪延遲。
- **儀表板**:Streamlit 加一個 tab,展示每次請求的 token/成本/延遲、檢索命中、三模式對比(把 `metrics.py` 的數據可視化)。

**深度體現**:接近生產的可觀測性與可靠性,而不是 print 大法。

**交付與驗收**:一次請求能在 tracing 面板完整回放;儀表板實時顯示成本與延遲。

## Day 7 — 打磨 · 封裝 · 敘事:Docker + CI + 文檔 + Demo

**目標**:把它「發布」出去,並講成一個好故事。

**具體任務**
- **封裝**:寫 `Dockerfile` + `docker-compose`;`.devcontainer` 已有,順勢完善一鍵起環境。
- **CI**:GitHub Actions 跑 lint + 測試 + 一個 smoke 評測(小樣本),PR 即見綠燈。
- **README 重寫成工程案例**:架構圖、關鍵設計決策與權衡、改造前後的評測數字對比、成本分析、「下一步我會做什麼」。
- **決策記錄(ADR)**:建 `docs/adr/`,把「為什麼選 Chroma 而非 FAISS」「為什麼加 rerank」這類選擇記錄下來——面試時直接能講。
- **面試彈藥**:一頁「系統設計」總結 + 一份把每個功能對應到能力項的講稿;錄一段短 demo / 截圖。

**深度體現**:能交付 + 會講故事——這是把「專案」變成「作品」的最後一步。

**交付與驗收**:`docker compose up` 一鍵可跑;CI 綠;README 讀完就能理解全貌與你的技術判斷。

## 優先級(時間不夠時的取捨)

若七天壓縮,按此順序保留最高性價比的深度:
1. **Day 2 + Day 3**(真向量庫 + 混合檢索 + 引用)——收益最大、最能拉開檔次。
2. **Day 5**(評測驅動)——讓所有改動可量化,面試最能講。
3. **Day 4**(真 LangGraph 編排)——技術深度信號最強。
4. Day 1 / Day 6 / Day 7 為工程完整度加分項。

## 一句話串起七天
> 先把地基修平(D1),換上真正的檢索基建(D2)並讓它檢得準、答得有據(D3);把智能體編排重寫成真正的狀態圖(D4);用嚴謹評測量化每一次進步(D5);讓系統看得見、扛得住(D6);最後封裝發布並講成一個好故事(D7)。
